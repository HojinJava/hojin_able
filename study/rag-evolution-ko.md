# 한국어 RAG, 어디까지 발전했는가

> 같은 데이터로 무엇이 달라지는지 — 단계별 기대효과

---

## 배경: RAG란 무엇인가

**RAG(Retrieval-Augmented Generation)** 는 LLM이 답변하기 전에 관련 문서를 먼저 찾아서 참고하게 만드는 방식입니다.

```
[기본 구조]
질문 → 관련 문서 검색 → LLM에 전달 → 답변
```

문서를 LLM 내부에 모두 학습시키는 대신, 필요할 때마다 꺼내 쓰기 때문에 **최신 정보 반영, 비용 절감, 출처 추적**이 가능합니다.

---

## 실험 설정

모든 단계에서 동일한 데이터셋과 질문을 사용합니다.

**데이터셋:** 회사 계약서 50건 + 기술 보고서 30건 (전부 한국어)

**고정 질문 3가지:**

| 구분 | 질문 | 난이도 |
|------|------|--------|
| Q1 | "납기 지연 패널티 기준이 뭔가요?" | 단순 — 한 문서 안에 답이 있음 |
| Q2 | "A사와의 계약에서 우리가 불리한 조항들은?" | 중간 — 여러 문서를 연결해야 함 |
| Q3 | "전체 계약서에서 반복되는 법적 리스크 패턴은?" | 복잡 — 전체 코퍼스를 종합해야 함 |

## LV1 — 기본 RAG

**설정:** 500토큰 고정 청크, 영어 기반 임베딩 모델(ada-002)

### 한국어에서 특히 취약한 이유

한국어는 같은 의미를 매우 다양한 형태로 표현합니다.

```
"납기 지연 패널티" 와 의미가 같은 표현들:

  "납기지연패널티"       ← 붙여쓰기
  "납기가 지연될 경우"   ← 조사 변형
  "지연 발생 시 위약금"  ← 완전히 다른 단어 조합
```

영어로 학습된 임베딩 모델은 이 세 표현을 **전혀 다른 개념**으로 인식합니다.
영어 문서였다면 Q1이 65% 정도 나오지만, **한국어는 시작점 자체가 낮습니다.**

### 결과

| 질문 | 정확도 | 이유 |
|------|:------:|------|
| Q1 납기 패널티 | **35%** | 표현 변형을 같은 개념으로 인식 못함 |
| Q2 A사 불리한 조항 | **15%** | 위 문제 + 조항이 여러 페이지에 분산 |
| Q3 전체 리스크 패턴 | **0%** | 구조적으로 불가능 |

---

## LV2 — 임베딩·검색 개선

**추가:** LaBSE 임베딩 + 형태소 분석 기반 청킹 + 하이브리드 검색(BM25+벡터)

### 핵심 변화 1: LaBSE 임베딩

LaBSE는 109개 언어를 **동일한 벡터 공간**에서 처리하도록 설계된 다국어 임베딩 모델입니다.

```
LaBSE 벡터 공간에서의 거리:
  "납기 지연 패널티"    ←→  "납기가 지연될 경우 위약금"   → 매우 가까움 ✅
  "납기 지연 패널티"    ←→  "지연 발생 시 배상금"         → 가까움 ✅

ada-002 벡터 공간에서의 거리:
  "납기 지연 패널티"    ←→  "납기가 지연될 경우 위약금"   → 멂 ❌
  "납기 지연 패널티"    ←→  "지연 발생 시 배상금"         → 매우 멂 ❌
```

### 핵심 변화 2: 형태소 분석 기반 청킹

```
고정 500토큰 청킹 (문장이 잘림):
  "...을 초과하는 경우 갑은 을에게 계약금액의 0.1%에 해당하"
  "는 지체상금을 청구할 수 있다. 단, 천재지변..."

형태소 분석 기반 청킹 (문장 단위 보존):
  "납기를 초과하는 경우 갑은 을에게 계약금액의 0.1%에
   해당하는 지체상금을 청구할 수 있다."
```

### 핵심 변화 3: 하이브리드 검색

- **BM25 (키워드):** "A사"처럼 정확히 일치해야 하는 고유명사에 강함
- **벡터 검색:** 의미적으로 유사한 표현 찾기에 강함
- **두 점수를 합산**해 상호 보완

### 결과

| 질문 | 정확도 | 변화 |
|------|:------:|:----:|
| Q1 납기 패널티 | **85%** | +50%p |
| Q2 A사 불리한 조항 | **60%** | +45%p |
| Q3 전체 리스크 패턴 | **0%** | 변화 없음 |

> **한국어 LV1→LV2 점프: 평균 +47%p**
> 영어 환경에서 같은 전환이 +20%p인 것과 비교해 **2배 이상** 효과가 큽니다.
> 한국어 환경에서 이 단계는 선택이 아닌 기본값입니다.

## LV3 — 지식 그래프

**추가:** LLM 기반 개체·관계 추출 → 그래프 DB 저장(Neo4j) → 그래프 탐색 검색

### 핵심 변화: 텍스트 조각이 아닌 관계로 저장

```
기존 방식 (텍스트 청크 저장):
  "A사는 2024년 3월 을로서 본 계약을 체결하였으며..."  ← 덩어리째 저장

지식 그래프 방식 (개체와 관계로 분해):
  A사(공급업체)
    → [을_역할] → 본계약
  본계약
    → [포함] → 지체상금조항
  지체상금조항
    → [비율] → 0.5%
    → [조건] → 납품지연
```

Q2처럼 "A사와 연관된 모든 조항"을 찾으려면 **A사 노드에서 그래프를 따라가면** 5개 문서에 흩어진 조항들을 모두 수집할 수 있습니다.

### 결과

| 질문 | 정확도 | 변화 |
|------|:------:|:----:|
| Q1 납기 패널티 | **87%** | +2%p (단순 질문은 LV2로 충분) |
| Q2 A사 불리한 조항 | **85%** | +25%p (여러 문서 조항 통합) |
| Q3 전체 리스크 패턴 | **35%** | 개별 문서 리스크 파악 가능, 패턴은 아직 약함 |

---

## LV3.5 — 온톨로지 자동화 *(독립 적용 가능)*

**추가:** 문서 샘플 분석 → 개체·관계 타입 자동 발견 → 멀티에이전트 정제 → 일관된 스키마 확정

### LV3과 LV3.5의 차이

LV3의 지식 그래프는 **사람이 직접 정의한 스키마**를 기반으로 추출합니다.

```
수동 스키마 (LV3):
  개발자가 직접 정의:
  {"entities": ["공급업체", "날짜", "금액"], "relations": ["체결", "포함"]}

  문제점:
  - "지체상금", "위약금", "배상금" → 각각 다른 개체로 저장 (실제론 같은 개념)
  - "을", "수급인", "공급자" → 각각 다른 개체로 저장 (실제론 같은 역할)
```

온톨로지 자동화는 **문서 자체에서 스키마를 발견**합니다.

```
자동 온톨로지 (LV3.5):
  문서 50건 샘플 분석
    → LLM: 반복 등장하는 개체·관계 타입 수집
    → Coverage Critic: "이 타입이 빠졌습니다"
    → Consistency Critic: "이 두 관계 타입이 충돌합니다"
    → 정제 루프 (최대 5회 반복)
    → 최종 스키마 확정

  결과:
  - "지체상금", "위약금", "배상금" → 손해배상조항(타입)으로 정규화
  - "을", "수급인", "공급자" → 계약당사자_을(타입)으로 정규화
```

### 독립 적용 가능한 이유

이 단계는 LV3 없이도 **LV2에 붙이는 것만으로 효과**가 있습니다.
일관된 메타데이터 태깅 → 검색 필터 정밀도 향상 용도로도 활용됩니다.

### 결과

| 질문 | 정확도 | 변화 |
|------|:------:|:----:|
| Q1 납기 패널티 | **90%** | +3%p (유사 표현 정규화 효과) |
| Q2 A사 불리한 조항 | **90%** | +5%p (같은 개념이 하나의 노드로 합쳐짐) |
| Q3 전체 리스크 패턴 | **50%** | +15%p (일관된 타입으로 패턴 인식 향상) |

## LV4 — 커뮤니티 감지 + 자동 모드

**추가:** Leiden 알고리즘 클러스터링 → 클러스터별 LLM 요약 → GLOBAL/LOCAL 자동 선택

### 핵심 변화: 전체 코퍼스를 주제별로 묶는다

```
50개 계약서 → Leiden 알고리즘 분석 →

  클러스터 A (12건): "납기·지연 관련 계약서"
    → LLM 요약: "납기 기준이 불명확한 계약 8건 포함, 지체상금 기준 0.1~0.5% 분포"

  클러스터 B (18건): "손해배상·위약금 관련 계약서"
    → LLM 요약: "손해배상 상한이 없는 계약 15건, 면책 조항 누락 계약 7건"

  클러스터 C (9건):  "지식재산권·기밀유지 관련 계약서"
    → LLM 요약: "기밀유지 기간 미명시 9건 전부, 퇴직 후 의무 기간 불일치"
```

Q3처럼 "전체 패턴"을 묻는 질문은 이 클러스터 요약들을 검색해 종합합니다.

### 자동 모드 선택

```
Q1 "납기 지연 패널티 기준이 뭔가요?"
  → LLM 판단: 특정 사실 질문 → LOCAL 모드 (그래프 탐색)

Q3 "전체 계약서에서 반복되는 리스크 패턴은?"
  → LLM 판단: 전체 합성 질문 → GLOBAL 모드 (커뮤니티 요약 검색)
```

### 결과

| 질문 | 정확도 | 변화 |
|------|:------:|:----:|
| Q1 납기 패널티 | **90%** | 유지 |
| Q2 A사 불리한 조항 | **90%** | 유지 |
| Q3 전체 리스크 패턴 | **82%** | +32%p (이전까지 불가능했던 질문이 가능해짐) |

## 어디서 시작해야 하나

```
내 상황                                추천 레벨
─────────────────────────────────────────────────
한국어 문서가 있다                  →  LV2 (무조건)
특정 회사/인물/계약 질문이 많다     →  LV3 추가
추출 품질이 들쭉날쭉하다            →  LV3.5 단독
전체 트렌드·패턴을 보고 싶다        →  LV4 추가
```

> **가장 빠른 ROI:** LV1 → LV2 전환.
> 설정 변경만으로 한국어 검색 정확도가 평균 2배 이상 오릅니다.
